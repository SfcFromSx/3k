# System Design Document: Three Kingdoms - Who Am I?

## 1. Introduction & Overview
This document serves as the technical blueprint for the "Three Kingdoms - Who Am I?" AI simulation game. It defines the application architecture, component structure, state management, and the specific interfaces required for building the client-side application using a modern tech stack.

## 2. Technology Stack
Given the requirement to interact purely with a local Ollama instance and require zero backend deployment for game logic, the application will be a 100% Client-Side SPA (Single Page Application).

- **Core Framework**: React 18+ with TypeScript (bootstrapped via Vite).
- **Styling**: TailwindCSS for layout, UI structuring, and responsive design. Custom CSS/SCSS will be used for the Chinese ink wash aesthetic (gradients, background textures).
- **State Management**: Zustand. Chosen for its lightweight, boilerplate-free approach to managing complex global states (Game Config, Turn Execution, Chat Logs).
- **LLM Communication**: Native `fetch` API directly querying `http://localhost:11434/api/`.
- **Icons & Assets**: Lucide React for UI icons; local JSON for the character database.

---

## 3. High-Level Architecture

The system is divided into three primary layers:
1.  **View Layer (React Components)**: The UI components that render the setup screen, the chat arena, player cards, and configuration panels.
2.  **State Layer (Zustand Stores)**: centralized stores that hold the active game state, player configurations, and the chat history.
3.  **Service Layer (AI & Game Engine)**: Asynchronous TypeScript services that handle the game loop logic, prompt construction, and HTTP requests to the Ollama API.

## 4. State Management (`store/`)

We will utilize two primary Zustand stores to manage concerns independently.

### 4.1. `useConfigStore.ts`
Manages the user's setup before a game begins.
```typescript
interface ObjectConfig {
  rounds: number;
  turnsPerRound: number;
  language: 'EN' | 'CN';
  translationEnabled: boolean;
  models: {
    playerA: string;
    playerB: string;
    playerC: string;
    judge: string;
    translator: string;
  };
}
```

### 4.2. `useGameStore.ts`
Manages the active game session, the current round, and the chat history.
```typescript
interface PlayerState {
  id: 'A' | 'B' | 'C';
  assignedCharacterId: string;
  turnsUsed: number;
  hasGuessed: boolean;
  score: number;
  skillFileStr: string; // Loaded from localStorage
}

interface ChatMessage {
  id: string;
  sender: 'System' | 'Judge' | 'PlayerA' | 'PlayerB' | 'PlayerC';
  textEN: string; // Original text generated by LLM
  textCN?: string; // Populated asynchronously if translation enabled
  timestamp: Date;
}

interface GameState {
  status: 'IDLE' | 'SETUP' | 'PLAYING' | 'ROUND_OVER' | 'MATCH_OVER';
  currentRound: number;
  activePlayerId: 'A' | 'B' | 'C';
  players: Record<string, PlayerState>;
  chatLog: ChatMessage[];
  // Actions
  dispatchTurn: (playerId: string, message: string) => void;
  evaluateWinCondition: () => void;
  // ...
}
```

## 5. Component Structure (`src/components/`)

### 5.1. `App.tsx` (Root)
Manages the high-level routing/conditional rendering between `SetupScreen` and `GameArena`.

### 5.2. `SetupScreen/`
- **`SetupForm.tsx`**: Form controls for Rounds, Turns, and Language.
- **`ModelSelector.tsx`**: Fetches `/api/tags` from Ollama on mount. Renders a dropdown assigning specific local models to Judge, Translator, A, B, and C.

### 5.3. `GameArena/`
- **`Header.tsx`**: Displays current Round, Status, and total UI points.
- **`PlayerCards/`**: 
    - `CardList.tsx`: Renders the three players.
    - `IdentityCard.tsx`: Displays the Player ID, chosen LLM name, their secret Character (with a "Hidden" eye-slash overlay for visual flavor), and current Score/Turns remaining.
- **`ChatInterface/`**:
    - `ChatLog.tsx`: The main scrollable feed.
    - `MessageBubble.tsx`: A polymorphic component based on `sender`. Handles distinct colors, avatars, and conditionally renders the inline `textCN` translation.
- **`ControlPanel.tsx`**: Auto-play speed toggle, pause execution, skip-to-end buttons.

## 6. Service Layer & The Game Engine (`src/services/`)

This layer contains the pure TypeScript orchestrators that do not render UI but execute the game rules.

### 6.1. `ollamaService.ts`
```typescript
export const fetchModels = async (): Promise<string[]> => { ... }
export const generateChatResponse = async (model: string, messages: {role: string, content: string}[]): Promise<string> => { ... }
```

### 6.2. `gameEngine.ts`
This is a state machine that drives the `useGameStore`.
1.  **`startRound()`**: Picks 3 distinct characters from `characters.json`. Initializes `PlayerState`. Pushes a System message to `chatLog`.
2.  **`executePlayerTurn(playerId)`**:
    -   *Build Prompt*: Gathers all generic history + Player's specific assigned roles + Player's specific `skill_file` from `localStorage`.
    -   *Await Ollama*: Calls `generateChatResponse()`.
    -   *Push Message*: Adds player's question to `chatLog`.
    -   *Trigger Translation* (Async background task if enabled).
3.  **`executeJudgeTurn(playerId, question)`**:
    -   *Build Prompt*: Explicitly restricts the Judge to the 3 permitted phrases regarding `playerId`'s assigned character.
    -   *Await Ollama*: Calls `generateChatResponse()`.
    -   *Update State*: Increments `turnsUsed` for the player.
4.  **`runRoundReview()`**: (The Learning Mechanic)
    -   Iterates through players. Takes the complete round log.
    -   Prompts the respective LLM: "Analyze your performance. What logic worked? What failed? Output a dense summary."
    -   Appends output to `localStorage.getItem('skill_A')`.

## 8. Data Flow: A Single Turn Execution

1.  **GameEngine** determines it is Player A's turn.
2.  **GameEngine** constructs Player A's Context Array (System Prompt + Skill File + Previous Chat Log).
3.  **Ollama API** receives the request for Model A. Model A returns a string: *"Am I affiliated with Wei?"*
4.  **GameStore** pushes this question to `chatLog`. UI updates.
5.  *(Concurrent)*: **TranslatorService** intercepts the new `chatLog` entry, hits the Translator Model, and patches `textCN` into the log object. UI re-renders inline translation.
6.  **GameEngine** constructs the Judge's array (Character: Zhao Yun + Rules: Yes/No ONLY + Context: Player A's question).
7.  **Ollama API** receives request for Judge Model. Judge returns: *"No."*
8.  **GameStore** pushes Judge's answer to `chatLog`. Updates Player A's `turnsUsed`. UI updates.
9.  **GameEngine** yields turn to Player B.
